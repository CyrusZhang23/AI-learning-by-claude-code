# 层5 — 多 AI · 多设备协同

> 可移植性：**中**。本层单机完成；"多设备"全部用云端 agent / 多进程 / 模拟方式，不需要第二台物理设备。

## 目标

让**不同的 AI 编码工具**在同一个项目上协作（Claude Code + Codex + Gemini/Aider），理解它们之间如何交接；并用模拟方式理解"跨设备"协同（定时 agent、远程触发、推送、云端 agent）。

## 你将亲手产出

- `workspace/lab-05/pipeline.sh` —— 一个多工具协作的流水线脚本
- `workspace/lab-05/handoff.md` —— AI 之间的"交接约定"文档（输入/输出契约）

## 前置检查

- [ ] 至少装了 Claude Code；**建议**再装一个：Codex CLI 或 Gemini CLI 或 Aider（没有也能用模拟方式学概念）
- [ ] 完成层3（MCP）—— 步骤4 会用到
- [ ] 建好 `workspace/lab-05/` 工作区，放一个小项目（如一个待优化的 Python 脚本）

## 分步教案

### 步骤 1 — 设计交接契约
- **做什么**：先写 `handoff.md`：约定"探索方"产出什么格式的报告、"实现方"消费什么、产物落在哪个文件。不同 AI 之间没有共享内存，**文件和 git 就是它们的通信总线**。
- **为什么**：多 AI 协作的第一原则是定好接口契约，否则交接会混乱。
- **通过标准**：你有一份清晰的"谁产出什么、谁消费什么"约定。

### 步骤 2 — 两工具协作（Claude + Codex/Gemini）
- **做什么**：让 Claude Code 做探索/规划，把结论写进 `plan.md`；再用另一个工具（`codex exec` 或 `gemini`）读 `plan.md` 去实现。若只装了 Claude，用两个独立 `claude -p` 会话模拟"两个角色"。
- **为什么**：体会异构 AI 通过文件交接的真实工作流，以及各工具的风格差异。
- **通过标准**：第二个工具产出的代码确实基于第一个工具的 plan。

### 步骤 3 — 三方流水线
- **做什么**：把流程串成 `pipeline.sh`：A 探索 → B 实现 → C 评审/测试（C 可以是第三个工具，或同工具的"评审角色"）。每段读上一段的产物文件。
- **为什么**：理解 AI 流水线 = 一串"读输入文件 → 产出输出文件"的阶段，可像 Unix 管道一样组合。
- **通过标准**：`bash pipeline.sh` 能一条龙跑完探索→实现→评审，产出三份阶段产物。

### 步骤 4 — MCP 作为 AI 间桥梁
- **做什么**：用层3 学的 MCP，把一个 AI 暴露成 MCP server（如 `codex --serve`，或你层3 写的 server 充当"专家工具"），让 Claude 通过 MCP **直接调用**另一个 AI，而不是靠文件交接。
- **为什么**：文件交接是异步松耦合；MCP 是同步紧耦合。理解两种 AI 间协作范式的取舍。
- **通过标准**：Claude 能通过 MCP 协议调到另一个 AI 并拿回结果。

### 步骤 5 — 跨设备协同（模拟）
- **做什么**：单机模拟"跨设备"的三种桥梁：(a) `CronCreate` 建一个定时 agent，模拟"另一台机器定时干活"；(b) 了解 `RemoteTrigger` / `PushNotification` 的作用；(c) 用云端 agent（claude.ai/code）跑一个任务，体验**真·跨设备**——你在浏览器/手机发起，云端执行，无需第二台本地机器。
- **为什么**：理解"多设备多 AI"的本质是**异步触发 + 远程执行 + 通知回流**，设备只是载体。
- **通过标准**：你能说清定时 / 远程触发 / 推送 / 云端 agent 各自在跨设备协同里扮演什么角色。

## 完成标志

- 你有一份多 AI 交接契约和一个能跑的多工具流水线
- 你至少体验过"文件交接"和"MCP 直连"两种 AI 间协作方式
- 你能讲清"跨设备协同"为什么本质上是异步触发+远程执行，而非真的需要多台设备
