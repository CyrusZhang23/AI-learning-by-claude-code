# 层6 — 多 AI · 多设备协同

> 可移植性：**中**。本层单机完成；"多设备"全部用云端 agent / 多进程 / 模拟方式，不需要第二台物理设备。

> 🎯 **本层一句话**：文件是异构 AI 的通信总线；干活方自报"行为等价"≠验收，必须独立只读裁判才算 —— 而**闸门的覆盖面 = 你显式列出的不变量 / 边界，不是模型强弱、也不是评审道数**。

## 目标

让**不同的 AI 编码工具**在同一个项目上协作（Claude Code + Codex + Gemini/Aider），理解它们之间如何交接；并用模拟方式理解"跨设备"协同（定时 agent、远程触发、推送、云端 agent）。

## 你将亲手产出

- `workspace/lab-06/pipeline.sh` —— 一个多工具协作的流水线脚本
- `workspace/lab-06/handoff.md` —— AI 之间的"交接约定"文档（输入/输出契约）

> 远程接入（Tailscale + 手机 SSH + tmux）的配置已前移到**层 1 文末"可选加餐"**——想从手机/平板上课的，先去那里配好再回来。

## 开始前：先确认你的"AI 兵力"（决定这一层怎么上）

> 这一层的核心是**多个 AI 协同**——不同的 AI 编码工具在同一个项目上接力交接。但**大多数学员手上只有 Claude Code 一个**，这完全不影响学习。
>
> **教练必做：先问学员装了哪些 AI CLI**（如 `codex` / `gemini` / `cursor-agent` / `aider`），据此选路：
>
> | 你的情况 | 这一层怎么上 |
> |---|---|
> | **有 ≥2 个 AI 工具** | 走"真异构"路线：用不同 vendor 跑步骤 2-4 的流水线，顺便体会不同 AI 的风格差异与交接 |
> | **只有 Claude Code** | 走"纯 Claude 模拟"路线：用**多个独立 `claude -p` 进程 + 各自文件夹/CLAUDE.md/settings.json** 扮演探索/实现/评审三角色，照样跑通整条流水线——概念 100% 一样，只是三棒都是 Claude。**不必为这门课特地去装一堆 AI** |
>
> 两条路要掌握的机制完全相同：文件交接契约、独立只读评审、打回重试。**有多 AI 是加分项，没有也能完整学完这一层。**

## 分步教案

### 步骤 1 — 设计交接契约
- **做什么**：先写 `handoff.md`：约定"探索方"产出什么格式的报告、"实现方"消费什么、产物落在哪个文件。不同 AI 之间没有共享内存，**文件和 git 就是它们的通信总线**。
- **为什么**：多 AI 协作的第一原则是定好接口契约，否则交接会混乱。
- **通过标准**：你有一份清晰的"谁产出什么、谁消费什么"约定。

### 步骤 2 — 两工具协作（Claude + Codex/Gemini）
- **做什么**：让 Claude Code 做探索/规划，把结论写进 `plan.md`；再用另一个工具（`codex exec` 或 `gemini`）读 `plan.md` 去实现。若只装了 Claude，用两个独立 `claude -p` 会话模拟"两个角色"。
- **为什么**：体会异构 AI 通过文件交接的真实工作流，以及各工具的风格差异。
- **通过标准**：第二个工具产出的代码确实基于第一个工具的 plan。

### 步骤 3 — 三方流水线
- **做什么**：把流程串成 `pipeline.sh`：A 探索 → B 实现 → C 评审/测试（C 可以是第三个工具，或同工具的"评审角色"）。每段读上一段的产物文件。
- **▶ 让学员先预测（本层最重要的一问）**：B 实现完，往往会自报"我测过了，行为完全等价"。问学员——"既然 B 自己说测过了，C 这道独立评审还有必要吗？" 很多人答"B 都说没问题了，C 多余"。**揭晓**：极有必要。实测里 B（无论是 Codex 还是 Claude）把 `find_duplicates` 改用 `Counter` 后，对 `NaN` 输入行为分叉（`[]`→`[nan]`），却照样自报"均等价"——被独立只读的 C 抓回。**干活方自报 ≠ 验收**。再追一问加深："C 凭什么比 B 可靠？是因为 C 用了更强的模型吗？"（答案：不是，是因为 C **上下文隔离 + 只读权限**，立场独立；同 vendor 同模型也行。）
- **为什么**：理解 AI 流水线 = 一串"读输入文件 → 产出输出文件"的阶段，可像 Unix 管道一样组合；而**质量靠独立只读闸门，不靠模型强弱**。
- **通过标准**：`bash pipeline.sh` 能一条龙跑完探索→实现→评审，产出三份阶段产物；学员能说清"为什么干活方不能给自己验收"。

### 步骤 4 — MCP 作为 AI 间桥梁
- **做什么**：用层4 学的 MCP，把一个 AI 暴露成 MCP server，让 Claude 通过 MCP **直接调用**另一个 AI，而不是靠文件交接。实测命令（codex）：`codex mcp-server` 以 stdio 起 server，再 `claude mcp add codex-expert -- codex mcp-server` 注册。注册后它在新会话里是 `mcp__codex-expert__codex` 工具（同 layer4 `mcp__<server>__<tool>` specifier）。
- **为什么**：文件交接是异步松耦合（落盘、对方自己读、留产物可审计）；MCP 是同步紧耦合（像调函数/RPC，阻塞等返回、零中间文件）。理解两种范式取舍：要持久可审计可重跑→文件；要推理中途一句即时专家意见→MCP。
- **通过标准**：Claude 能通过 MCP 协议调到另一个 AI 并拿回结果。
- **🪤 坑（▶ 先让学员预测）**：在**当前**会话里跑完 `claude mcp add codex-expert -- codex mcp-server` 之后，问学员——"现在这个会话里能立刻调到 `mcp__codex-expert__codex` 这个工具吗？" 多数人答"刚加完当然能"。**揭晓：不能**。MCP server 在**会话启动时加载**；已开的会话不会热加载新 server，要**新起一个 `claude`/`claude -p`** 才有。（同源机制：层 2 的自定义斜杠命令、层 5 的自定义 agent 也都是新会话才加载。）

### 步骤 5 — 跨设备协同（模拟）〔进阶·可选〕
> 这一层重点是**多 AI 协同**（步骤 1-4）。跨设备只是延伸：大多数学员单机即可，没有多设备/没连 Remote Control 的，**把下面几种桥的"角色"读懂即可，不必都跑通**。
- **做什么**：单机模拟"跨设备"的三种桥梁：(a) `CronCreate` 建一个定时 agent，模拟"另一台机器定时干活"；(b) 了解 `RemoteTrigger` / `PushNotification` 的作用；(c) 用云端 agent（claude.ai/code）跑一个任务，体验**真·跨设备**——你在浏览器/手机发起，云端执行，无需第二台本地机器。
- **为什么**：理解"多设备多 AI"的本质是**异步触发 + 远程执行 + 通知回流**，设备只是载体。
- **通过标准**：你能说清定时 / 远程触发 / 推送 / 云端 agent 各自在跨设备协同里扮演什么角色。
- **🪤 两个实测坑（关于 PushNotification 推到手机）**：
  1. **要会话启动时就开 Remote Control**：推送到手机/iPad 走 Claude 的 Remote Control 通道（≠ ssh/mosh/tmux，那只是终端传输层，跟推送无关）。必须用 `claude --remote-control` 启动会话才开；普通 `claude` 启动的会话 `remoteEnabled=false`，推送到不了手机。
  2. **终端有焦点时推送被抑制**：你正盯着终端看时，`PushNotification` 会返回 `Not sent — terminal has focus`——设计如此（你在场就不打扰）。要验证手机推送，得制造"你真的走开了"的失焦场景。
- **机制小结**：`CronCreate`/`RemoteTrigger`=触发，本地会话/云端=执行，`PushNotification`=回流；三者凑齐才是"跨设备"，设备本身只是载体。

### 步骤 6 — 全景扫描（收尾）[核心]
- **做什么**：对照本层两种协作范式问学员——① "文件交接 vs MCP 直连，各自什么时候用？"（持久可审计可重跑→文件；推理中途要一句即时专家意见→MCP）；② 跨设备三角色 `CronCreate`/`RemoteTrigger`=触发、本地/云端=执行、`PushNotification`=回流，哪几个本层跑通了、哪几个只读了概念。再确认那条最硬的洞察："流水线质量上限在'边界列得多全'，不在加几道评审。"
- **为什么**：把"文件 vs MCP""触发/执行/回流"两组对照固化成索引，避免只记住"跑通了 pipeline.sh"。

> **🧪 留个回归夹具**：`pipeline.sh`（A→B→C 串联 + 打回重试循环 + `MAX_ITERS` 封顶）本身就是夹具。关键是用**哨兵行 `VERDICT:PASS`** 作 C 与脚本的机器可读握手（别 grep 自然语言"通过"，会误命中）。改教案后重跑，确认第 2 轮能自动收敛（B实现→C打回→B修→PASS）。
